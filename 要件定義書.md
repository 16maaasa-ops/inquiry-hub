# 要件定義書：マルチチャネル問い合わせ集約 + AI分類システム

作成日: 2026-07-16（同日更新: テストデータに合わせてカテゴリを不動産向け4分類に確定）
ステータス: 承認済み
想定顧客: 不動産会社（賃貸・売買仲介）

---

## Why：なぜ作るのか

- 月 750 件の問い合わせがメール / LINE / 電話に分散しており、管理しきれていない
- 営業時間外の LINE 問い合わせを取りこぼしている（推定取りこぼし率 15%）
- クレームへの気づきが遅く、認知まで平均 4 時間かかっている

**目標**: 取りこぼしゼロ、クレーム認知を 5 分以内にする。

## Who：誰が使うのか

| 利用者               | 使い方・状況                                                                     |
| -------------------- | -------------------------------------------------------------------------------- |
| 問い合わせ対応チーム | 日中、Slack のカテゴリ別チャネルを見て問い合わせを確認・対応する                 |
| 営業部長             | クレーム発生時に個人 LINE で通知を受け取る。外出中・営業時間外もスマホで気づける |

※ 管理画面を使う「管理者」ロールは MVP では想定しない（管理画面は作らない）。

## What：何ができればゴールか

### 必須機能（MVP）

1. **Gmail → Slack 集約**: 問い合わせメールを自動で Slack に流す
2. **LINE → Slack 集約**: LINE 公式アカウントへのメッセージを自動で Slack に流す
3. **AI による 4 カテゴリ分類**: Claude API で分類し、カテゴリ別 Slack チャネルへ振り分ける
   - カテゴリ（不動産業務に合わせた 4 分類。テストデータ `data/case5-test-inquiries.csv` の期待カテゴリに準拠）:
     「賃貸」「売買」「内見」「クレーム」
   - 上記に当てはまらない無関係なメッセージ（雑談・スパム等）は「その他」として受け止め、取りこぼさずに専用チャネルへ流す（誤ってクレーム扱いしない＝ノイズ耐性）
4. **クレームの LINE 即時通知**: クレーム判定から **5 分以内** に営業部長の個人 LINE へ Push 通知

### 後回しにする機能（Phase 2 以降）

- 電話の音声テキスト化（Twilio 連携）
- SLA レポートの自動生成
- 管理画面（履歴一覧などの Web 画面）

## How：技術構成

キューを挟んだ信頼性重視の構成を採用する。処理が一度失敗してもキューに残るため再処理でき、「クレームを絶対に取りこぼさない」要件に合う。

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   Gmail      │    │  LINE 公式   │    │   電話       │
│  (Webhook)   │    │  (Webhook)   │    │  (Phase 2)   │
└──────┬───────┘    └──────┬───────┘    └──────────────┘
       │                   │
       └─────────┬─────────┘
                 │
                 ▼
         ┌──────────────────┐
         │ Vercel Functions  │
         │ - 受信            │
         │ - 整形            │
         │ - キューに投入    │
         └────────┬─────────┘
                  │
                  ▼
         ┌──────────────────┐
         │ Supabase Queue    │
         │ + Cron(Headless)  │
         └────────┬─────────┘
                  │
       ┌──────────┴──────────┐
       ▼                     ▼
┌──────────────┐      ┌──────────────┐
│  Claude API   │      │  通常通知     │
│ (カテゴリ分類)│      │  → Slack      │
└──────┬───────┘      └──────────────┘
       │
       ▼ (クレーム判定なら)
┌──────────────┐
│ LINE Push     │
│ (個人 LINE)   │
└──────────────┘
```

- **フレームワーク**: Next.js（API Routes / Vercel Functions）、パッケージ管理は npm
- **ホスティング**: Vercel
- **データ保存・キュー**: Supabase（Queue + Cron で非同期処理）
- **AI 分類**: Claude API
- **外部連携**: Gmail API(Webhook) / LINE Messaging API / Slack API

### 想定コスト（提案書より）

- 初期費用: 14.8 万円（税込）
- 月額: 約 1.5 万円（Vercel Pro + Supabase Free + LINE/Slack/Claude API 使用量）

## Look：見た目・雰囲気

### Slack 通知（メインの「画面」）

リッチなカード形式。カテゴリ・経路・送信者・本文が一目でわかり、操作ボタン付き。

```
🚨 クレーム発生（左端に赤バー・@channel 通知）
━━━━━━━━━━━━━━
🔴 未対応
📨 経路: LINE
👤 送信者: 山田太郎 様（LINE表示名）
🕒 受信: 7/16 21:45
━━━━━━━━━━━━━━
先月入居した部屋のエアコンが効きません。
至急対応してください。苦情です。
━━━━━━━━━━━━━━
[ 担当になる ] [ 対応済にする ]
```

通常カテゴリはヘッダーの絵文字で見分ける: 🏠賃貸 / 💰売買 / 📅内見 / 📮その他

## テストデータ

- `data/case5-test-inquiries.csv`（22件）を分類精度の検証に使用する
- 列構成: 番号 / チャネル(line・mail) / 問い合わせ本文 / 期待カテゴリ / 緊急フラグ / 検証する機能
- 含まれる検証観点: 各カテゴリの分類精度、クレーム2件の緊急判定（5分以内LINE Push）、無関係メッセージのノイズ耐性、「緊急ではありません」という文言での誤検知防止

### 将来の管理画面（Phase 2 以降）

信頼感重視のビジネス系（白ベース + ネイビー / ブルー）。

## 成功指標

- 取りこぼし率: 推定 15% → **0%**
- クレーム認知遅延: 平均 4 時間 → **5 分以内**
- 納期: 5 営業日
